起步

为了逼自己学习，有一天起床后，就直接买了个苹果开发者账号。起步完全是靠iOS Programming(4th Edition)了。边看书，边照着把代码都打了一遍。书里面的好多原理部分还没弄懂，找机会得重看一遍。

动手

看完书后，就不知天高地厚的觉得入门了…于是动手开始做这个简洁的目标管理App了。因为书里面数据处理用的是core data，然后界面大部分都是用的xib，所以动手的时候也是那么做的。后来，数据变复杂了，就不知道如何用core data处理了…因为确实没太弄明白博大精深的core data。最后，完全放弃了core data，直接用FMDB和sqlite来处理数据了，界面方面也全部用storyboard弄了。从那开始，进度就快了很多。但是每周最多只抽出6个小时来弄，慢慢的就过了4个月了。过程中还是学到了很多的，每每掉到坑里，总是可以求助Stack Overflow爬出来。

结束

最后，谢谢V2EX的各位大神。每天在这逛，心里总是痒痒的想做点啥…现在又开始学习Web开发了，发现有太多的东西要学了！MongoDB，Express，Ember，Node…已经忙不过来了。

新年，新的开始

Posted on 2015-02-04   |  

已经二月份了，我才开始写这个题。其实很正常，因为我好像什么事都是反应慢半拍。

去年

  • 学习了iOS开发。看完了iOS Programming: The Big Nerd Ranch Guide。这本书虽然有点旧了，但是内容还是讲的很清楚的。有大量的图片来描述程序运行时发生了什么，对了解底层的知识很有帮助。不足的是没有怎么介绍Storyboard的用法。对于个人开发来说，感觉还是Storyboard要方便很多，而且未来趋势应该也是Storyboard主导吧。（这是我瞎说的） 我自己反正全面转向Storyboard来做App了。

背景

今天做面试题目。题目的要求还很好满足，10分钟不到就写完了。但是纠结data type的时候花了好久，最后越来越紧张，直到最后被迫提交答案了都还没解决…

纠结data type其实就是因为题目有个要求是，计算的结果如果大于2147483647就return -1。

纠结

其实主要的纠结就是int a和int b相乘，结果会overflow。于是就想着把相乘的结果cast到double，然后和2147483647比大小判定。

然后就囧了，不知道当时怎么整的！怎么test结果都不对…其实就是这么一行code…double c = (double)a * b

总结

现在想来真丢脸啊，还是不应该没睡醒就做题啊…以后要记住了

int_32和int_32相乘，还是int_32，丢掉了前面的32个比特！要解决这个问题，cast其中一个int_32到一个64位的数就可以了…gcc会自动的达到我们想要的32*32->64！

你们知道前面那个纠结我纠结了多久么？50分钟…答案提交后我5分钟就找到答案了，我真的太容易不淡定了！

Let’s start with something simple: an immutable stack.

Now, immediately I hear the objection: a stack is by its very nature something that changes. A stack is an abstract data type (ADT) with the interface

You push stuff onto it, you pop stuff off of it, it changes. How can it be immutable?

Every time you need to make a data structure immutable, you use basically the same trick: an operation which “changes” the data structure does so by constructing a new data structure. The original data structure stays the same.

How can that possibly be efficient? Surely we’ll be allocating memory all over the place! Well, actually, in this case, no. An immutable stack is every bit as efficient as a mutable stack. Even better: in some cases, it can be considerably more efficient, as we’ll see.
